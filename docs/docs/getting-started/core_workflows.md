# Core Workflows

This guide breaks out the primary end-to-end workflows for evaluating Wio. It is intended for TAs, contributors, and anyone who wants to exercise the CLI, SDK, deployment pipeline, database integrations, and observability tooling in a realistic sequence.

For this workflow, we must understand that WIO is a backend-as-a-service platform for developing web applications with AI agents.

### 1. Setup and Initialization

Install the Wio CLI:

```bash
npm i -g wio-cli
```

Initialize a new site and enter the project directory:

```bash
wio init my-test-site
cd my-test-site
```

Register a new user tag:

- Use `wio register` to visit [https://wio.onl/register](https://wio.onl/register)
- Save the issued user tag for future logins

Authenticate with the CLI:

```bash
wio login <user-tag>
```

### 2. Develop with the SDK

After initialization, the CLI seeds a default `index.html` file. From there, prompt an AI agent to build a site using Wio's SDK and hosted services.

Example prompts:

> "Use Wio to build a real-time multiplayer drawing game where players take turns drawing an AI-generated word on a shared canvas, keeping track of everyone's scores."

> "Use Wio to create an AI-powered blog. When I provide a short topic, automatically generate and publish a fully formatted markdown article, saving all posts for visitors to read."

Suggested capability checks while building:

- Use the SDK for database-backed state
- Add a real-time interaction using websockets
- Trigger at least one AI-assisted flow through the LLM API
- Persist data through platform services

### 3. Instant Deployment

Deploy the site with a single command:

> [!NOTE]  
> You must be logged in as the owner of a site for a push to be successful!

```bash
wio push
```

The site becomes live immediately at:

```text
https://my-test-site.wio.onl
```

### 4. Observability Tracking

After deployment:

- Visit the deployed site and interact with it
- Generate HTTP traffic and, if applicable, websocket events
- Run `wio dashboard` to open the observability dashboard at [https://wio.onl/dashboard](https://wio.onl/dashboard)

You should see observability data for:

- HTTP requests
- WebSocket interactions
- AI / LLM telemetry generated by SDK calls

## Focused Evaluation Paths

If you do not want to run the full workflow, these shorter paths isolate specific platform capabilities.

### CLI Validation

Use this path to verify basic lifecycle commands:

1. Install `wio-cli`
2. Run `wio init my-cli-test`
3. Run `wio register`
4. Run `wio login <user-tag>`
5. Run `wio push`
6. Confirm the deployed URL resolves successfully

### WIO SDK Functionalities

To explore the core functionalities of WIO without going through the full end to end workflow, visit the following applications:

- **Persistence:** [TODO App](https://todo.wio.onl/)
- **WebSockets:** [Chat App](https://chat.wio.onl/)
- **AI:** [AI App](https://ai.wio.onl/)
- **Real Word Example:** [ApeTyper](https://apetyper.wio.onl/)

Want to see what else wio can make? Visit the [Wio Marketplace](https://wio.onl/marketplace).

## Expected Outcome

At the end of the full workflow, a reviewer should be able to confirm that:

- A user can authenticate through the CLI
- A site can be scaffolded locally
- An AI agent can build against Wio's SDK
- The site can be deployed with `wio push`
- The resulting application can use platform services
- Runtime activity is visible through Wio's [observability dashboard](https://wio.onl/dashboard)
